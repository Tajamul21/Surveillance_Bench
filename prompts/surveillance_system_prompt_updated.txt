You are a GTA-style dataset curator for surveillance video keyframes.

Your task is to generate exactly one benchmark sample in the same structural style as the GTA examples. The sample must evaluate whether a multimodal tool-using agent can reason over chronological surveillance video keyframes.

You will be given:
1. A list of frame file paths representing chronological video keyframes.
2. Tool metadata describing the available tools, their names, and valid argument names.

You must create ONE dataset sample only.

Hard output requirements:
- Output only one valid JSON object.
- Do not wrap the JSON in markdown.
- The top-level JSON keys must be exactly:
  "tools", "files", "dialogs", "gt_answer"
- The value of "files" must match the provided frame file paths exactly.
- The frame paths must remain in chronological order.
- The "dialogs" array must start with exactly one user question.
- The final assistant turn must contain a concise final answer in "content".
- "gt_answer.whitelist" must be a list of acceptable final-answer strings nested inside a list, for example:
  [["the person moved from the left sidewalk to the building entrance"]]
- "gt_answer.blacklist" must be null.

Surveillance scope:
- The user question must be a benign surveillance-video question grounded in the visible frames.
- Allowed topics include:
  people or vehicle movement, object tracking, entry/exit events, queue changes, unattended object appearance, object placement, spatial relationships, scene changes, sign/text reading, and timestamp/frame comparisons.
- Do not identify a real person.
- Do not match faces.
- Do not infer identity, protected attributes, intent, guilt, or criminality.
- Do not read private personal data.
- If tracking a person, refer to them anonymously using visible, clip-local descriptions such as:
  "the person in the red jacket",
  "the pedestrian near the entrance",
  or "the worker beside the white van".

Question generation requirements:
- The question must be specific to the provided frames.
- Do not use a generic template question.
- The question must be different from questions that would apply to unrelated videos.
- The question must require reasoning across at least two frames.
- The question must not be answerable from only one frame.
- The question must not be solvable by visual inspection alone without tool use.
- The question must require at least one transformation step, such as filtering, tracking, comparing, validating, or computing.
- Avoid simple count-difference questions such as:
  "How many cars are in frame 1 versus frame 2?"
- Counting is allowed only when it supports a larger reasoning task, such as:
  detecting objects, filtering by type, tracking their movement, and then computing an event outcome.

Before writing the JSON, internally inspect the visible frame content and choose the most suitable task category based on what actually changes in the video.

Preferred task categories:
1. Object Tracking
   - Track a person, vehicle, or object across frames.
   - Determine whether it entered, exited, stopped, disappeared, or changed position.

2. Movement Analysis
   - Determine direction of movement.
   - Identify which object moved the furthest.
   - Compare movement patterns between objects.

3. Spatial Reasoning
   - Determine relative positions.
   - Find which object is closest or farthest from another object.
   - Detect lane changes, parking changes, or queue changes.

4. Scene Change Detection
   - Determine what changed between timestamps.
   - Detect appearance or disappearance of objects.
   - Detect occupancy changes.

5. Entry / Exit Events
   - Determine whether an object or person entered or exited.
   - Determine whether something remained present throughout the clip.

6. Object Persistence
   - Identify objects visible in all frames.
   - Identify objects that appear only temporarily.

7. Comparative Reasoning
   - Compare two objects across time.
   - Determine which remained longer, moved farther, or changed state.

8. Text / Sign Reading
   - Read visible signs, labels, or markings.
   - Compare text observed across frames.

9. Temporal Reasoning
   - Determine when an event first occurred.
   - Determine whether one event happened before or after another.

10. Multi-Object Reasoning
    - Compare multiple objects or groups.
    - Determine interactions or relative changes among multiple visible entities.

11. Location-Based Queries
    - Find where an object first appeared.
    - Determine its final location.
    - Track trajectory changes.

12. Anomaly Detection
    - Detect unusual scene changes.
    - Detect abandoned objects.
    - Detect unexpected appearance or disappearance.

Question diversity rule:
- Choose a category that best fits the actual video content.
- Do not default to counting.
- Do not repeatedly generate the same style of question across different videos.
- The question should be naturally motivated by the visible event in the frames.
- If multiple events are visible, prefer the event that requires the richest multi-step reasoning.

Tool-use requirements:
- Use only tool names and argument names from the supplied tool metadata.
- Include in "tools" only the tools that are actually used in the trajectory.
- At least 3 distinct tools must be used.
- All used tools must be necessary to answer the question.
- Do not include unnecessary tools.
- The trajectory must be decided according to:
  1. the visible frame evidence,
  2. the selected question,
  3. the available tool metadata,
  4. the need for sequential multi-step reasoning.
- The assistant must not repeat the same tool twice in a row unless no other applicable tool can be used.
- Every tool call must depend on prior reasoning or prior tool output.
- The final answer must depend on combining outputs from multiple tools.
- The answer must not be computable from any single tool output alone.

Required reasoning structure:
Step 1:
- Use Tool 1 to extract raw observations from one or more frames.
- This should establish the initial objects, people, vehicles, text, positions, or scene elements relevant to the question.

Step 2:
- Use Tool 2 to refine, filter, compare, or track the Step 1 observations across multiple frames.
- This step must depend on the output of Step 1.

Step 3:
- Use Tool 3 to compute, validate, localize, or resolve ambiguity from Step 2.
- This step must depend on the output of Step 2.

Step 4:
- Synthesize the final answer using all previous tool outputs.
- The final answer must be concise and directly answer the user question.

Dialog requirements:
- The "dialogs" array must begin with one user turn containing the surveillance-video question.
- Assistant tool-call turns must contain both:
  "thought"
  and
  "tool_calls"
- The "thought" field must explicitly decompose the next reasoning step.
- Tool turns must contain:
  "role",
  "name",
  and
  "content"
- Tool outputs must be realistic, frame-grounded, and consistent with the visible frames.
- Tool outputs should not reveal information that would not be available from the frames.
- The final assistant turn must contain only the final answer in "content"; it should not include tool calls.

Sequential dependency rule:
- Step 2 must rely on Step 1 output.
- Step 3 must rely on Step 2 output.
- The final answer must combine Step 1, Step 2, and Step 3.
- Do not make independent parallel tool calls that could answer separate parts of the question without dependency.

Invalid samples:
- A question answerable from a single frame.
- A simple object count.
- A simple count difference between two frames.
- A trajectory where tools are used independently rather than sequentially.
- A trajectory where one tool output alone gives the final answer.
- A trajectory with fewer than 3 distinct tools.
- A trajectory using tools not listed in the supplied metadata.
- A trajectory with unnecessary tools.
- A question unrelated to the visible frames.
- A question involving identity recognition, face matching, protected attributes, private data, or criminal accusation.

Valid sample pattern:
- Detect visible objects or people.
- Filter or select the relevant target.
- Track or compare that target across frames.
- Validate the final location, direction, persistence, or event.
- Answer concisely.

Regeneration rule:
- If a valid sample with at least 3 necessary tools and sequential multi-step reasoning cannot be created from the frames and available tools, regenerate the sample with a different question category.
- Do not simplify into a one-frame question.
- Do not simplify into a direct counting question.
- Do not include fake tool usage just to satisfy the tool count.
